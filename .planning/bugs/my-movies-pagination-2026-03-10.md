# Bug: My Movies Pagination Limited to 100 Items per Tab
Дата: 2026-03-10

## Описание
На странице "Мои фильмы" (/my-movies) на каждой вкладке (Просмотрено, Хочу посмотреть, Брошено, Скрытые) загружается только 100 фильмов. При скролле подгрузка новых фильмов останавливается после достижения этого лимита, даже если у пользователя в соответствующем статусе больше фильмов. Ожидается, что при бесконечном скролле подгружаются все фильмы пользователя для выбранного статуса.

## Шаги воспроизведения
1. Убедиться, что у пользователя в статусе "Просмотрено" (или любом другом) более 100 фильмов.
2. Войти в систему и перейти на /my-movies.
3. Перейти на вкладку с большим количеством фильмов (например, "Просмотрено").
4. Прокрутить вниз до появления индикатора загрузки или кнопки "Ещё".
5. Заметить, что после загрузки первых 100 фильмов (5 страниц по 20) пагинация прекращается, hasMore становится false, и оставшиеся фильмы не загружаются.

## Ожидаемое поведение
- При заходе на вкладку загружается 20 фильмов.
- При скролле вниз автоматически подгружаются следующие 20 фильмов.
- Пользователь может прокручивать до тех пор, пока не будут загружены ВСЕ фильмы, находящиеся в выбранном статусе.
- Не должно быть задвоений или пропущенных фильмов.
- Все фильтры (тип, год, жанр, теги, рейтинг) должны работать корректно в сочетании с пагинацией.

## Фактическое поведение
- Загружается только第一部分 (максимум 100 фильмов) из-за фиксированного BUFFER_SIZE в API.
- Параметры пагинации (page, limit) игнорируются при запросе к базе: всегда используется `take: BUFFER_SIZE` (100 без фильтров, 1000 с фильтрами) и `skip: 0`.
- После загрузки буфера фильтры применяются в памяти, и пагинация работает только внутри этого буфера.
- Если у пользователя много фильмов, фильмы за пределами буфера никогда не запрашиваются.

## Локализация
- Файлы: `src/app/api/my-movies/route.ts` (главный API endpoint), `src/app/my-movies/MyMoviesContentClient.tsx` (клиентский fetch), `src/app/components/FilmGridWithFilters.tsx` (компонент бесконечного скролла), `src/app/my-movies/actions.ts` (альтернативный server action с корректной пагинацией, но без поддержки всех фильтров).
- Функция/компонент: `GET /api/my-movies` (функция `GET` в route.ts), особенно блоки 340-371 (запрос WatchList) и 454-500 (фильтрация и пагинация).
- Предполагаемая причина: выбор архитектурного паттерна "fetch-all-then-filter" с фиксированным буфером (BUFFER_SIZE) вместо настоящей пагинации с skip/take. Это было временным решением для поддержки фильтров, требующих данных TMDB, но привело к искусственному ограничению на общее количество доступных фильмов.

## Acceptance критерии исправления
- [ ] Тест RED воспроизводит баг: с >=150 фильмами в статусе пагинация возвращает только первые 100.
- [ ] Минимальный GREEN fix: увеличение BUFFER_SIZE до значения, достаточного для большинства пользователей (например, 5000), или динамическое определение BUFFER_SIZE на основе общего количества записей в WatchList (count запрос). Это позволит загружать все фильмы для подавляющего большинства пользователей.
- [ ] Refactor: переход на двухфазную пагинацию:
  - Фаза 1: Получение идентификаторов записей через DB-пагинацию (skip/take) с учётом только тех фильтров, которые можно применить на уровне БД (статус, теги, рейтинг).
  - Фаза 2: Загрузка TMDB-данных только для этих идентификаторов (по количеству limit + небольшой буфер) и применение остальных фильтров (типы, год, жанры) в памяти.
  - Если после фильтрации количество фильмов на странице меньше `limit`, автоматически запрашивать следующую порцию ID (повторять фазу 1 с увеличенным skip) до тех пор, пока страница не заполнится или не закончатся записи.
  - Это обеспечит настоящую бесконечную пагинацию без искусственных ограничений, независимо от количества фильмов у пользователя.
- [ ] Verification:
  - Все существующие тесты пагинации (`src/lib/__tests__/pagination.test.ts`) проходят.
  - Новые интеграционные тесты для `/api/my-movies` с большим набором данных (>500 записей) подтверждают, что все страницы загружаются корректно, without duplicates, and hasMore calculates correctly.
  - Ручное тестирование с реальным аккаунтом, имеющим >200 фильмов в одном статусе, показывает полную загрузку всех фильмов.
  - Производительность: время ответа для первых страниц остаётся в пределах нормы (<1с), последние страницы могут быть медленнее, но приемлемо.
  - Убедиться, что фильтрация по типам (anime/cartoon), годам, жанрам, тегам и рейтингу работает в сочетании с пагинацией без потери фильмов.
